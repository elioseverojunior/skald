# Phase 4a Baseline: Full Competitor Benchmark Results

**Date:** 2026-06-03
**Branch:** `worktree-feat+beat-noyalib`
**Purpose:** Establishes the measurement baseline for Phase 4b optimization work.

---

## Environment

| Field | Value |
|-------|-------|
| Toolchain | `rustc 1.93.1 (01f6ddf75 2026-02-11)` |
| Platform | `Darwin 24.6.0 x86_64` (Intel Mac, native x86\_64 — NOT Apple Silicon) |
| CPU | Intel Xeon W-2150B @ 3.00 GHz (Skylake-W, 10 cores, AVX-512-capable) |
| System load at run time | 4.05 / 7.45 / 8.27 (1/5/15 min) — moderately loaded; see caveats |

### Measurement settings

```text
--warm-up-time 2  --measurement-time 8
```

Criterion default 100 samples was used; Criterion extended collection time automatically
where needed to meet the sample target. Times reported as the median (middle value of the
`[low median high]` bracket). The group `measurement_time` floor in the bench source is
15 s (serialize) or 15 s (deserialize); Criterion honoured the larger of the two.

### Commands used

```bash
# Serde (deserialize + serialize, all libs, all fixtures)
cargo bench -p skald-bench --bench serde \
    -- --measurement-time 8 --warm-up-time 2 \
    2>&1 | tee /tmp/bench_serde.txt

# Tree-level roundtrip (skald vs yaml_rust2 vs rust_yaml)
cargo bench -p skald-bench --bench end_to_end \
    -- --measurement-time 8 --warm-up-time 2 roundtrip_compare \
    2>&1 | tee /tmp/bench_e2e.txt
```

## Excluded competitor: `serde_yml`

`serde_yml 0.0.13` is **excluded** from all benchmarks.

- **RUSTSEC-2025-0068** — unsound: a Serializer use-after-free can trigger a segfault in safe Rust.
- The upstream repository is archived; no fix is expected.
- Skald's `deny.toml` enforces `vulnerability = "deny"` with zero ignored advisories.

A commented-out entry in `skald-bench/Cargo.toml` documents this exclusion in-source.

## Group 1: `serde_deserialize`

Deserializes each YAML fixture into the library's generic value type
(`skald::Value`, `noyalib::Value`, `serde_yaml_ng::Value`, `serde_json::Value`
for serde\_saphyr which has no native Value type). Throughput is measured against
the YAML byte length; `serde_json` operates on the JSON-equivalent fixture
(speed-of-light reference, not a YAML comparison).

| Library | small (218 B) | medium (~5 KB) | large (~121 KB) | Notes |
|---------|--------------|----------------|-----------------|-------|
| **skald** | **30.66 µs** | **318.1 µs** | **17.72 ms** | This project |
| noyalib | 49.43 µs | 232.3 µs | 10.79 ms | Target to beat |
| serde\_yaml\_ng | 33.43 µs | 343.4 µs | 18.82 ms | |
| serde\_saphyr | 57.85 µs | 571.7 µs | 33.51 ms | Into `serde_json::Value` |
| serde\_json (JSON) | 3.94 µs | 49.51 µs | 3.09 ms | Speed-of-light (JSON, not YAML) |
| serde\_yml | — | — | — | Excluded: RUSTSEC-2025-0068 |

**Verdict (small):** Skald ranks **2nd**. noyalib is ~1.6x faster on small inputs.
Skald edges out serde\_yaml\_ng by ~8%; both are well ahead of serde\_saphyr.

**Verdict (medium):** Skald ranks **2nd**. noyalib is ~1.4x faster on medium inputs.
Skald is ~7% faster than serde\_yaml\_ng.

**Verdict (large):** Skald ranks **2nd**. noyalib is ~1.6x faster on large inputs.
Skald is ~6% faster than serde\_yaml\_ng.

**Overall:** noyalib consistently leads deserialize by ~1.4–1.6x across all sizes. Skald holds
a solid #2 position. The gap appears consistent, pointing to algorithmic rather than
fixture-size-specific overhead — likely a hot path in the scanner or value construction.

## Group 2: `serde_serialize`

Serializes a pre-built value (deserialized outside the timed loop) back to a YAML string.
`serde_saphyr` serializes a `serde_json::Value` (its value was obtained via
`serde_saphyr::from_str` into `serde_json::Value`; this is the same data path used in
the deserialize group — apples-to-apples for serde\_saphyr).

| Library | small (218 B) | medium (~5 KB) | large (~121 KB) | Notes |
|---------|--------------|----------------|-----------------|-------|
| **skald** | **8.50 µs** | **81.93 µs** | **3.70 ms** | This project |
| noyalib | 10.35 µs | 62.80 µs | 5.37 ms | |
| serde\_yaml\_ng | 13.81 µs | 185.1 µs | 10.44 ms | |
| serde\_saphyr | 14.99 µs | 136.0 µs | 6.62 ms | Via `serde_json::Value` |
| serde\_json (JSON) | 1.03 µs | 7.43 µs | 344 µs | Speed-of-light (JSON) |
| serde\_yml | — | — | — | Excluded: RUSTSEC-2025-0068 |

**Verdict (small):** Skald ranks **1st** among YAML libraries. Skald is ~1.2x faster than noyalib.

**Verdict (medium):** Skald ranks **1st** by a wide margin. Skald is ~1.3x faster than noyalib
and ~2.3x faster than serde\_saphyr. serde\_yaml\_ng is the slowest at 185 µs.

**Verdict (large):** Skald ranks **1st** among YAML libraries. Skald is ~1.45x faster than noyalib.

**Overall:** Skald leads serialize across all fixture sizes. The emitter's design (trusting the
caller's `ScalarStyle`, no re-quoting overhead) pays off here.

## Group 3: `roundtrip_compare` (tree-level)

Full load+dump round-trip at the AST level. No serde involved.
`skald` = `from_str_node` + `to_string_node`; `yaml_rust2` uses
`YamlLoader::load_from_str` + `YamlEmitter::dump`; `rust_yaml` uses
`YamlLoader::load_from_str` + `YamlEmitter::new(..).dump`.

| Library | small (218 B) | medium (~5 KB) | large (~121 KB) | Notes |
|---------|--------------|----------------|-----------------|-------|
| **skald** | **28.65 µs** | **424.4 µs** | **13.05 ms** | This project |
| yaml\_rust2 | 33.49 µs | 414.1 µs | 15.08 ms | |
| rust\_yaml | 77.07 µs | 1.50 ms | 705.5 ms | |

**Verdict (small):** Skald ranks **1st** — ~14% faster than yaml\_rust2, ~2.7x faster than rust\_yaml.

**Verdict (medium):** Skald ranks **2nd** — statistically tied with yaml\_rust2 (within noise bounds);
both are ~3.5x faster than rust\_yaml.

**Verdict (large):** Skald ranks **1st** — ~13% faster than yaml\_rust2, ~54x faster than rust\_yaml.

**Overall:** Skald leads roundtrip at the tree level. yaml\_rust2 is a close competitor on medium
inputs. rust\_yaml is significantly slower, particularly on large inputs (705 ms vs 13 ms — ~54x).

## Caveats

- **System load:** The machine was under moderate load (1-min load average 4.05 on an
  Intel x86\_64 macOS host). Numbers are indicative but may have higher variance than
  a quiesced machine. Phase 4b should re-run benches on a lightly loaded machine.
- **Prior runs unreliable:** Project memory notes that earlier benchmark runs under
  very high load (64+ on 20 cores) were unreliable. This run is better but not idle.
- **ARM baseline deferred:** The plan called for an optional Linux aarch64 cross-arch
  data point via `ssh raspi5`. The Pi has no Rust toolchain yet; bootstrapping Rust
  and building six competitor crates on 4 Cortex-A76 cores was estimated too slow for
  the 4a sign-off time budget. ARM baseline is planned as a Phase 4b activity (NEON
  path validation on `ssh raspi5` is also needed for the `simd` feature gate).
- **serde\_saphyr serializer note:** serde\_saphyr has no native Value type; both
  its deserialize and serialize rows use `serde_json::Value` as the data carrier.
  This adds marginal JSON-side overhead but ensures consistent comparison within the group.
- **Criterion change% lines:** Criterion reported change% relative to a previous cached
  baseline for some runs (prior run left in target/criterion). These do not affect
  the absolute `time: [lo median hi]` values reported above, which are fresh measurements.
